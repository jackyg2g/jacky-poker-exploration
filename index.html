Technical Architecture Brief

Platform brief for a proprietary exploit system built on three strategy layers.

This platform is centered on a single modeling thesis: strong exploit strategy should not be generated as an isolated black box. It should be built by aligning three layers of intelligence: a GTO baseline, observed population tendencies, and a proprietary exploit strategy that intentionally departs from baseline where the field is predictably imbalanced. The surrounding system exists to ingest data, train those layers, version the outputs, and serve them as a usable product.

Core Thesis

Three strategy layers drive the entire platform.

Everything in the repository maps back to a three-layer strategy system: GTO baseline for equilibrium reference, population tendencies for observed field behavior, and proprietary exploit strategy for monetizable directional adjustment. The parser, training pipeline, API, and product interface all exist to produce, compare, and operationalize these three layers.

Three strategy layers

The central idea in the platform is the disciplined separation of strategy into three interpretable layers. This is what makes the system explainable, versionable, and commercially meaningful.

Layer 01

GTO baseline

The GTO layer is the equilibrium reference model. It provides the anchor against which all observed field behavior is measured. Without this layer, the system would know what the population does, but not whether that behavior is strategically weak or strong.

  • Purpose: define the reference policy for each supported spot.
  • Source: solver artifacts, curated references, and batch-generated baseline outputs.
  • Role: the baseline that keeps exploit adaptation grounded instead of arbitrary.
Layer 02

Population tendencies

The population layer is the empirical observation engine. It converts raw hand-history data into spot-level action frequencies, sample sizes, and confidence signals. This is the layer that tells the system how the field actually behaves rather than how theory says it should behave.

  • Purpose: map real-world tendencies by spot.
  • Source: parsed hand histories aggregated into canonical frequency distributions.
  • Role: the evidence base for exploit opportunity discovery.
Layer 03

Proprietary exploit strategy

The proprietary layer is the actual productized edge. It is generated by measuring where the population departs from GTO, filtering those deviations through significance and confidence rules, and then applying bounded exploit adjustments to produce a release-ready strategy package.

  • Purpose: turn field miscalibration into controlled strategic edge.
  • Source: deviation analysis plus adjustment and validation logic.
  • Role: the monetizable layer that differentiates the platform from a baseline viewer.
Why this matters

The system is valuable because it preserves the relationship between theory, evidence, and edge.

Many systems can show a baseline or summarize observed behavior. This platform is stronger because it formalizes the transformation from baseline, to population signal, to proprietary exploit package. That relationship is what makes the stack coherent.

  • GTO baseline answers: what is theoretically sound?
  • Population tendencies answer: what does the field actually do?
  • Proprietary exploit answers: how should strategy shift to capitalize on that gap?
  • The rest of the platform exists to compute, validate, release, and serve this progression.

Parser layer

Rust-based ingestion for high-volume hand histories

The parser processes raw text histories in parallel, emits normalized CSVs, and produces a schema suitable for PostgreSQL ingestion. This gives the platform a performant raw-data foundation before any strategy logic runs.

Training layer

Population, deviation, proprietary strategy, validation

The training stack converts real-world behavior into structured population tendencies, measures divergence from GTO, generates exploitative adjustments, and validates the resulting strategy before deployment.

Serving layer

Low-complexity delivery with version-aware lookups

FastAPI and `EngineService` load active artifacts into memory, making the current strategy package instantly queryable by the product interface and the CLI.

Strategy generation lifecycle

The training path is deliberately structured as a repeatable release pipeline. Each stage transforms the representation of the strategy system: from raw hands, to structured frequencies, to exploitable opportunity, to proprietary policy, to versioned deployment.

01
Load or build the GTO baseline

The system starts from `solver/gto_baseline.json`, with the solver layer available to generate or enrich that baseline using PioSOLVER or curated reference data.

02
Compute population tendencies

Parsed hand histories are aggregated into `spot_key -> action frequencies` with sample sizes and confidence measures, producing the platform’s view of observed field behavior.

03
Measure deviations from baseline

GTO and population distributions are compared spot by spot, with statistical significance and EV opportunity estimates used to identify the highest-value exploit windows.

04
Generate proprietary exploit strategy

Significant deviations are converted into proprietary action-frequency shifts through bounded, normalized exploit rules governed by confidence and aggression parameters.

05
Validate through Monte Carlo simulation

Proprietary and GTO policies are compared against population tendencies to estimate aggregate EV gain, holdout concordance, and degenerate strategy risk.

06
Deploy as a versioned release

Each strategy package is written as `vN/strategy.json` plus metadata, diffs, and an active pointer, creating a release history that the product can browse and compare.

Training internals

The training layer is not a single script wrapped around a few heuristics. It is a staged transformation pipeline with separate responsibilities for population extraction, deviation discovery, exploit generation, validation, and release production. Each stage changes the representation of the strategy from raw observations to a deployable package.

Population modeling

Observed behavior is aggregated into spot-level frequency distributions.

`training/population.py` consumes parser outputs when full CSVs are available and falls back to weaker aggregate analysis sources when they are not. Its role is to transform hand-level action sequences into canonical spot-level distributions keyed by `spot_key`.

  • Inputs: `hands.csv`, `players.csv`, `actions.csv`, or lower-fidelity aggregated analysis artifacts.
  • Recognition targets: preflop RFI, defense versus RFI, blind defense, flop c-bets, responses to c-bets, turn barrels, and later-street continuations.
  • Outputs: `action_freqs`, `sample_size`, and confidence-style metadata for each generated spot.
  • Role in the system: this is the bridge from raw behavior data into modelable strategy distributions.
spot_key -> { action_freqs, sample_size, confidence }
Deviation analysis

Population is compared against GTO to identify exploitable divergence.

`training/deviation.py` compares the baseline layer and the observed population layer across their shared spots. The objective is not only to detect that behavior differs, but also to rank where that difference is both statistically meaningful and economically relevant.

Delta computation

Each action frequency is compared action-by-action to produce a signed deviation vector.

Statistical test

A chi-squared test is applied to estimate whether the observed behavioral shift is significant.

Classification

Dominant patterns are labeled into categories such as over-folding, over-calling, under-raising, or generic sizing error.

Economic signal

Each deviation receives an EV opportunity estimate that determines exploit priority.

Proprietary strategy generation

Exploit policy is derived by controlled displacement away from GTO.

`training/adjustment.py` is the heart of the proprietary layer. It does not train a neural network and it does not solve a fresh equilibrium online. Instead, it takes statistically credible population deviations and transforms them into exploitative frequency shifts.

prop_freq = gto_freq + alpha * delta * confidence_weight
  • `alpha`: controls how aggressively the system leans from GTO toward exploit.
  • `delta`: captures how far the field is drifting from the baseline.
  • `confidence_weight`: scales the adjustment by statistical reliability and sample size.
  • Post-processing: action frequencies are clamped, normalized, and annotated with EV gain estimates and reasoning.
Over-folding -> more aggression Over-calling -> less bluff density Under-raising -> more pressure
Validation

Every candidate strategy package is stress-tested before release.

`training/validation.py` runs a simplified Monte Carlo process to compare the proprietary policy against the GTO baseline when both are played into the observed population tendencies. The goal is not to claim perfect game-tree evaluation, but to reject obviously weak or unstable policy changes before deployment.

  • Holdout split: spots are divided into training and holdout sets to reduce overfitting to the same observations.
  • Simulation: hero and villain actions are sampled from their respective action-frequency distributions.
  • Payoff model: a simplified payoff matrix estimates relative EV outcomes under each policy pairing.
  • Outputs: aggregate EV gain, average EV gain per spot, holdout concordance, and degenerate strategy flags.

Pydantic models define a stable contract across product surfaces.

`GameState`, `ActionFrequencies`, `StrategyResponse`, `PopulationSummary`, `VersionInfo`, and `DeviationReport` are shared concepts across API responses, tests, CLI output, and frontend rendering. This keeps the online product path strict and predictable.

EngineService

The runtime core is in-memory, version-aware, and artifact-backed.

On startup, the service loads the GTO baseline, population tendencies, and the active proprietary strategy package from disk. During request handling, it resolves the active package in memory rather than recomputing strategies online.

Runtime internals

The online service is intentionally simple in topology, but not simplistic in behavior. Its primary job is to turn a `GameState` into a stable `StrategyResponse` by resolving the correct spot, loading aligned layers, and returning a coherent comparison payload.

Engine lookup behavior

The service resolves three aligned strategic perspectives for every query.

For each request, `EngineService` first builds a canonical `spot_key`. It then loads the corresponding GTO frequencies, observed population frequencies, and active proprietary frequencies. The response is assembled as a single joined structure so the product never has to reconcile mismatched sources itself.

  • GTO layer: baseline strategic frequencies for the matched spot.
  • Population layer: observed field action frequencies plus sample size.
  • Proprietary layer: active exploit package frequencies plus EV gain estimate.
  • Reasoning: a deterministic narrative built from the largest observed deviations and proprietary shift.
Fallback and continuity

The product remains usable even when live infrastructure is incomplete.

The backend itself includes default and demo strategy fallbacks, and the frontend mirrors the API types while providing a demo mode when the backend is unavailable. That makes the product robust for presentation, design iteration, and isolated frontend development.

Backend fallback

Missing or malformed JSON data falls back to demo records so the service remains callable.

Frontend fallback

If API requests fail, the web client promotes itself into demo mode and still renders product flows.

Operational effect

The system can be demonstrated and iterated as a product even before all production dependencies are formalized.

Postgres and Redis are provisioned as scale layers, not yet the live center.

Solver and baseline generation

The solver layer is upstream of online serving. Its purpose is to build and evolve the GTO baseline rather than to make the production path depend on live solve latency.

Reference mode

Curated GTO reference data ensures the system can operate without local solver access.

`solver/gto_reference.py` contains prebuilt strategic reference spots. This gives the platform a portable baseline generation fallback and makes development, validation, and product iteration possible even on environments where PioSOLVER is not installed.

PioSOLVER mode

UPI integration supports deeper baseline expansion when solver infrastructure is available.

`solver/upi_client.py` wraps PioSOLVER’s stdin/stdout protocol. Combined with `batch_solver.py`, the platform can configure trees, apply board and bet-size settings, solve prioritized spots, checkpoint progress, and export resulting strategies into baseline artifacts.

  • Tree setup: pot, board, effective stack, ranges, and sizing trees are configurable.
  • Batch solving: spots are solved one at a time with checkpoint support.
  • Fallback behavior: if solving fails, the system drops back to reference mode rather than halting the pipeline.

Rust core and ingestion layer

The Rust layer is where the platform starts to look like infrastructure rather than application code. It handles scale-sensitive text parsing, structured extraction, and reusable low-level strategy abstractions.

Parser

Large hand-history corpora are converted into structured tables at high throughput.

`core/parser` can traverse directories of `.txt` hand histories, split them into single hands, parse header, seats, actions, board progression, and summary lines, and then stream normalized outputs into CSVs suitable for analytical processing.

  • Outputs: `hands.csv`, `players.csv`, `actions.csv`.
  • Concurrency: parallel batch processing is built in.
  • Database bridge: the parser emits PostgreSQL COPY schema and load statements automatically.
Engine abstractions

StrategyStore and spot primitives provide a path toward faster native serving.

`core/engine` includes action-frequency containers, three-layer strategy records, spot encoding, and nearest-neighbor lookup logic. These pieces are not yet the primary serving runtime, but they are exactly the kinds of primitives needed if the platform moves more of its hot path into Rust.

Strategy object anatomy

The entire platform revolves around a canonical strategy object keyed by spot and represented as aligned action-frequency distributions. This common data shape is what lets solver outputs, population observations, proprietary policy, API schemas, CLI tables, and the web product all remain interoperable.

Spot identity

`spot_key` reduces a game situation to a stable lookup address.

The system does not identify a query by full raw hand history. It extracts the decision-relevant state into a compact canonical key that can be reused across offline artifacts and online lookups.

nlhe_cash:{positions}:{stack_depth}:{board_texture}:{action_sequence}
  • Positions: hero and villain positional context such as BTN, SB, BB, CO.
  • Stack depth: effective stack abstraction in big blinds.
  • Board texture: simplified board-state encoding for postflop matching.
  • Action sequence: compressed representation of betting history up to the current decision.
Strategy payload

Every layer ultimately resolves to action frequencies plus operating context.

The API response combines three aligned action distributions and attaches the metadata required to interpret them. That includes sample size on the population side, EV lift and version info on the proprietary side, and a reasoning layer that explains why the exploit package departs from baseline.

{ spot_key, gto: { action_freqs }, population: { action_freqs, sample_size }, proprietary: { action_freqs, ev_gain_estimate, version }, reasoning }

Operational benefits

Versioning is doing more than naming snapshots.

  • Auditability: every deployed strategy package has metadata and a place in a historical sequence.
  • Comparability: the product can show release-to-release changes in coverage and EV lift.
  • Safety: an active pointer makes rollback a file-level change instead of a retraining event.
  • Governance: release notes and metadata form the beginning of a formal promotion discipline.
  • Portability: artifact packages can be moved across environments without requiring an online retrain.
