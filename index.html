Technical Architecture Brief

Three-layer poker intelligence, served as a product platform.

Holdem Exploration combines hand-history parsing, GTO baseline generation, population modeling, exploitative proprietary strategy training, versioned deployment, API serving, and product delivery into a single architecture. The result is a research-to-product system designed for repeatable strategy development rather than one-off analysis.

Current Operating Model

Artifact-backed serving with a clear path to scale.

The production path today is built around versioned strategy artifacts, in-memory serving, and a lightweight product interface. The surrounding infrastructure already anticipates the next layers: persistent storage, cache-backed distribution, and deeper solver integration.

System architecture

The architecture is designed as a continuous pipeline that begins with raw hand histories, produces structured population and strategy artifacts, and serves those artifacts through product interfaces. Each layer is separable, but the full value emerges when they operate as one loop.

Parser layer

Rust-based ingestion for high-volume hand histories

The parser processes raw text histories in parallel, emits normalized CSVs, and produces a schema suitable for PostgreSQL ingestion. This gives the platform a performant raw-data foundation before any strategy logic runs.

Training layer

Population, deviation, proprietary strategy, validation

The training stack converts real-world behavior into structured population tendencies, measures divergence from GTO, generates exploitative adjustments, and validates the resulting strategy before deployment.

Serving layer

Low-complexity delivery with version-aware lookups

FastAPI and `EngineService` load active artifacts into memory, making the current strategy package instantly queryable by the product interface and the CLI.

Strategy generation lifecycle

The training path is deliberately structured as a repeatable release pipeline. Each stage transforms the representation of the strategy system: from raw hands, to structured frequencies, to exploitable opportunity, to proprietary policy, to versioned deployment.

01
Load or build the GTO baseline

The system starts from `solver/gto_baseline.json`, with the solver layer available to generate or enrich that baseline using PioSOLVER or curated reference data.

02
Compute population tendencies

Parsed hand histories are aggregated into `spot_key -> action frequencies` with sample sizes and confidence measures, producing the platform’s view of observed field behavior.

03
Measure deviations from baseline

GTO and population distributions are compared spot by spot, with statistical significance and EV opportunity estimates used to identify the highest-value exploit windows.

04
Generate proprietary exploit strategy

Significant deviations are converted into proprietary action-frequency shifts through bounded, normalized exploit rules governed by confidence and aggression parameters.

05
Validate through Monte Carlo simulation

Proprietary and GTO policies are compared against population tendencies to estimate aggregate EV gain, holdout concordance, and degenerate strategy risk.

06
Deploy as a versioned release

Each strategy package is written as `vN/strategy.json` plus metadata, diffs, and an active pointer, creating a release history that the product can browse and compare.

Runtime serving model

The online product path is intentionally lightweight. The platform serves strategy artifacts from disk-backed version outputs rather than running solver computation on demand. That keeps response behavior simple, deterministic, and operationally efficient.

Platform stack

The repo is not a single application; it is a layered platform composed of ingestion, training, serving, and product surfaces. Some layers are already in the critical path, while others are scaffolded for the next phase of scale.

Core stack
  • RustHand-history parsing, batch processing, CSV generation, schema emission
  • PythonTraining pipeline, FastAPI service layer, strategy orchestration, CLI
  • Next.jsProduct interface for strategy exploration, analysis, population review, and release visibility
Primary runtime source
  • ArtifactsVersioned JSON strategy packages are the live serving substrate today
  • EngineServiceLoads active package into memory and exposes consistent lookup and reporting methods
  • PydanticDefines stable contracts between service, tests, CLI, and frontend
Scale foundation
  • PioSOLVERIntegrated at the solver layer for deeper baseline generation when available
  • PostgresProvisioned in infrastructure and aligned with parser outputs for formalized data persistence
  • RedisProvisioned for cache and future workload distribution when serving moves beyond artifact-only paths
Current architecture posture

Research-grade depth with product-grade delivery surfaces.

The platform already unifies data ingestion, strategy construction, release management, and user-facing interfaces. In its current form, it favors low-complexity, high-clarity serving through artifacts, while keeping the system open for deeper persistence, caching, and solver-backed expansion.

  • Artifact-backed serving keeps online complexity low while preserving release discipline.
  • Versioned strategy packages create a concrete history of coverage, gain, and change.
  • Rust, Python, and web layers are already partitioned by role rather than blended into a monolith.
  • Database, cache, and richer runtime execution paths are prepared as scaling layers rather than speculative placeholders.
